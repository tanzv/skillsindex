# Skill 详情页导航与预览优化设计

日期：2026-03-18

## 背景

当前 `frontend-next` 的公共 Skill 详情页已经具备基础信息展示、安装方式、资源浏览、版本历史、相关技能与互动能力，但整体仍存在两个核心问题：

1. 顶部站点导航与 Skill 内部导航没有形成清晰的分层
2. 预览区更像普通内容块，而不是页面的主工作台

对照参考页可以看到，用户在进入详情页后会自然同时关注三件事：

1. 这是什么 Skill
2. 值不值得继续看
3. 如果现在要用，安装入口和资源入口在哪里

当前页面虽然覆盖了这些信息，但它们的组织方式仍偏“内容堆叠”，没有形成稳定的阅读路径与交互秩序。

本次优化采用折中方向：

1. 保留 SkillsIndex 现有品牌与公共市场风格
2. 吸收参考页在信息架构与预览驱动交互上的优点
3. 不把页面改造成完全类 IDE 的三栏工作台

## 目标

在不改变公共路由、数据契约与总体页面语义的前提下，将 Skill 详情页升级为“站点导航浅、Skill 上下文深、预览工作台重”的详情结构：

1. 明确区分站点级导航与 Skill 级导航
2. 让用户在首屏同时完成价值判断、安装判断与阅读入口定位
3. 让预览区成为页面中心，而不是附属内容块
4. 保持安装动作始终可达，但不压过主预览区
5. 为后续增加深链、资源筛选、预览工具条等能力预留扩展点

## 非目标

本次不做以下事项：

1. 不调整 `/skills/[skillId]` 路由结构
2. 不引入新的后端接口或修改现有公共 schema
3. 不新增三栏 IDE 级文件管理布局
4. 不改造评论、收藏、评分的业务规则
5. 不引入与现有公共市场风格冲突的独立视觉语言
6. 不扩大到分类页、搜索页、排行榜页的整体重构

## 设计原则

### 1. 站点导航做浅，Skill 导航做深

全局顶栏继续承担站点级跳转职责，例如品牌、分类、排行榜、搜索、登录与主题/语言切换。
Skill 详情内部导航不再混入站点导航表达，而是沉到详情页自身的上下文层中。

### 2. 预览工作台是页面中心

详情页的核心不只是展示元数据，而是帮助用户进入 Skill 内容本身。
因此预览区必须具备稳定边界、稳定头部、稳定滚动容器与稳定切换心智。

### 3. 安装动作始终可达，但不抢第一视觉

安装方式、复制操作与源码入口需要持续存在，但它们的职责是服务决策和执行，而不是替代预览成为主舞台。

### 4. 同一页面，不同 Tab，应共享同一交互骨架

`Overview`、`Install`、`SKILL.md`、`Resources`、`Related`、`History` 允许内容不同，但不应在结构与节奏上像完全不同页面。
切换应保留稳定容器、稳定标题区和稳定主次关系。

### 5. 改结构优先于改皮肤

本次首先解决信息组织、状态边界和交互稳定性问题。
视觉表现应沿用现有公共市场 token 和 Skill Detail 样式体系，不扩散为一次无边界的重做。

## 信息架构设计

### 第一层：站点级顶栏

保留现有公共市场顶栏职责：

1. 品牌入口
2. 分类与排行榜等站点级主入口
3. 搜索、登录、主题与语言切换

这一层在 Skill 详情页中不再承载 Skill 内部内容切换，只负责“离开当前 Skill，去其他市场路径”。

### 第二层：Skill 上下文条

在顶栏下方增加独立的 Skill 上下文条，作为详情页的第二层导航。
该层由三部分组成：

1. 面包屑
2. Skill 上下文摘要
3. 内容导航

#### 面包屑

当前面包屑从弱提示升级为稳定上下文入口，目标是让用户知道自己正处于：

1. 市场首页
2. Skill 详情层级
3. 当前 Skill

#### Skill 上下文摘要

在上下文条中集中表达对决策最重要的状态，例如：

1. 作者或来源主体
2. 分类与子分类
3. 更新时间
4. 质量分
5. 收藏、评分、评论规模

这些信息不取代 Header，而是作为滚动后仍然可见的“当前对象状态”。

#### 内容导航

将当前 Tab 从普通 pills 升级为明确的 section navigation，覆盖：

1. `Overview`
2. `Install`
3. `SKILL.md`
4. `Resources`
5. `Related`
6. `History`

当用户滚动离开 Header 后，这一层仍然保留当前位置与可切换入口。

## 页面布局设计

### 整体布局

桌面端保持双栏结构：

1. 左中主栏：价值摘要 + 主预览工作台
2. 右栏：安装方式、复制动作、源码入口、互动操作

右栏继续采用吸顶策略，但其视觉强度低于主工作台。

### Header 角色

Header 保留页面首屏的对象识别职责，重点回答：

1. Skill 叫什么
2. 来源是什么
3. 属于哪个分类
4. 大致质量如何
5. 适合继续往下看吗

Header 下方不再承担完整内容导航职责，导航重心移交给上下文条。

### 主预览工作台

主栏在 Header 之后进入统一的工作台区域。
工作台的目标不是简单换内容，而是为不同 Tab 提供共同骨架：

1. 稳定的头部区域
2. 稳定的预览容器
3. 稳定的滚动逻辑
4. 稳定的操作位

## 导航与交互设计

### 吸顶上下文条

上下文条在进入详情内容区后保持吸顶，并包含：

1. 左侧：面包屑与当前 Skill 简称
2. 中部：Tab 导航
3. 右侧：当前预览对象状态，例如当前文件名、资源数量或版本数量

作用如下：

1. 避免滚动后失去当前上下文
2. 减少用户反复回到首屏寻找入口
3. 让导航和预览状态同步表达

### Tab 升级为工作区切换

本次不把 Tab 视为单纯的按钮组，而是视为工作区切换器。
切换后的要求如下：

1. 主工作台容器尽量保持稳定高度和边界
2. 标题、说明和状态信息保留一致结构
3. 不同 Tab 内容差异存在，但交互节奏一致

### 预览区统一化

`Overview`、`SKILL.md` 与 `Resources` 应尽量共享一套预览舞台。
统一预览区需要具备：

1. 文件或内容标题
2. 类型或语言信息
3. 更新时间
4. 可扩展动作位
5. 固定可感知高度与内部滚动

这样可以把“阅读内容”变成稳定体验，而不是每个区域都重新学一遍界面。

### Resources 模式

`Resources` 模式升级为“列表驱动预览”：

1. 资源列表负责选择当前文件
2. 主预览区负责渲染当前文件内容
3. 切换文件时更新当前高亮、预览头与预览内容
4. 不强制整页回到顶部

该模式下即使资源为空，也应保留工作台骨架，而不是让主区域塌缩为普通空白段落。

### Install 模式

安装信息分为两层：

1. 右侧吸顶栏中的快速执行层
2. 主工作台中的完整说明层

具体策略：

1. 任意 Tab 下右侧都保留关键安装入口
2. 切到 `Install` Tab 时，主栏展示完整安装说明与上下文解释
3. 安装动作不要求用户必须离开当前阅读上下文

## 组件与代码结构设计

### 页面容器

继续以 `frontend-next/src/features/public/PublicSkillInteractiveDetail.tsx` 作为页面级容器，统一收口详情页状态与回调。

### 新增组件一：SkillDetailContextBar

新增一个独立上下文导航组件，建议命名为：

- `SkillDetailContextBar`

职责：

1. 渲染第二层导航
2. 展示面包屑
3. 展示当前 Skill 状态摘要
4. 渲染 Tab 导航
5. 渲染当前预览对象状态

不负责：

1. 数据获取
2. 安装交互
3. 评论交互
4. 资源内容加载

### 新增组件二：SkillDetailPreviewStage

新增统一预览舞台组件，建议命名为：

- `SkillDetailPreviewStage`

职责：

1. 承接预览头
2. 承接预览内容容器
3. 提供统一空状态与加载态位置
4. 为未来预留动作 slot，例如复制、全屏、换行切换、跳源码

### 现有组件职责调整

#### PublicSkillInteractiveDetail

继续作为页面入口和状态拥有者，负责：

1. `workspaceTab`
2. `resourceContent`
3. `busy`
4. `feedback`
5. `commentDraft`
6. 后续新增的 `selectedResourceName`
7. 后续新增的 `previewMode`

#### SkillDetailWorkbench

继续承担 Tab 内容装配职责，但要减轻它对预览结构的直接定义。
它应转为：

1. 根据 Tab 选择面板
2. 将数据传给统一预览舞台
3. 保持各面板的外部结构一致

#### SkillDetailSidebar

继续负责安装和互动，但增强它与主工作台之间的弱联动，例如：

1. 当前 Tab 状态
2. 当前安装 audience
3. 当前资源模式提示

它不应接管主预览区状态。

## 状态归属设计

### 页面级状态

核心状态继续收口在 `PublicSkillInteractiveDetail`：

1. `workspaceTab`
2. `resourceContent`
3. `busy`
4. `feedback`
5. `commentDraft`

### 新增状态建议

如果实现时需要增强交互，仅建议新增：

1. `selectedResourceName`
2. `previewMode`

原因如下：

1. 路由容器拥有“用户当前在看什么”的真状态
2. workbench 只消费状态并触发回调
3. 预览组件不隐藏业务状态
4. 后续同步到 URL、增加最近浏览文件、增加资源筛选时，扩展路径清晰

## 扩展点

本次设计为以下后续能力预留了稳定扩展点：

### 1. URL 深链

后续可以将以下状态同步到 query string：

1. `tab`
2. `file`

### 2. 资源浏览增强

后续可以在不改预览舞台结构的前提下增加：

1. 文件搜索
2. 按语言过滤
3. 按更新时间排序

### 3. 预览工具条

后续可增加：

1. 复制内容
2. 全屏展开
3. 换行开关
4. 跳转源码链接

### 4. 新 Tab 扩展

后续如需新增：

1. `Discussions`
2. `Changelog`
3. `Governance`

只需要扩展 Tab 配置与 panel mapping，不需要重做页面骨架。

## 假设

本次设计建立在以下假设之上：

1. 不修改后端 schema，继续复用 `detail`、`resources`、`versions`、`resourceContent`
2. 不引入新的全局 token，优先复用现有 marketplace 与 skill-detail CSS 变量
3. 不改变公共路由结构，仍使用 `/skills/[skillId]`
4. 不把页面升级为三栏 IDE 工作台，继续保持“主内容 + 吸顶侧栏”的基础版式

## 测试与验证策略

### 单元测试

至少需要补强以下契约：

1. 第二层导航正确渲染当前上下文
2. Tab 切换后结构稳定，关键容器未丢失
3. `Resources` 模式下当前文件与预览头正确同步
4. 相关技能链接继续兼容 prefixed public route
5. 预览舞台在无资源或无内容场景下保留稳定空状态

优先补强的测试文件：

1. `frontend-next/tests/unit/skill-detail-workbench.test.ts`
2. `frontend-next/tests/unit/marketplace-topbar-slots.test.ts`
3. 如新增组件，补充对应组件测试

### 命令验证

本次实现完成后至少执行：

1. `cd frontend-next && npm run test:unit`
2. `cd frontend-next && npm run build`

如果实现过程中扩大到明显的交互重排，再补充：

1. `cd frontend-next && npm run test:e2e`

## 风险

1. 吸顶上下文条可能与现有顶栏和 breadcrumb 产生重复表达，需要在结构层消除冲突
2. 预览舞台统一后，`Overview` 与 `Resources` 的边界如果处理不好，可能出现“内容更统一但信息更平”的问题
3. 右侧安装栏继续吸顶时，若视觉强度过高，会重新压制主预览区
4. 如果状态没有收口在页面容器中，后续增加 URL 深链时容易出现状态分叉

## 结论

本次 Skill 详情页优化应优先解决“导航分层不清”和“预览舞台不够中心”这两个结构问题，而不是停留在局部换色或单点样式修饰。

推荐实现方向是：

1. 保留站点顶栏作为浅层站点导航
2. 新增 Skill 上下文条作为深层详情导航
3. 将主栏重构为统一预览工作台
4. 保持右侧安装与互动栏持续可达但不过度抢焦点

这样可以在不破坏现有公共市场风格和路由契约的前提下，让详情页真正具备“可判断、可安装、可阅读”的完整工作流。
