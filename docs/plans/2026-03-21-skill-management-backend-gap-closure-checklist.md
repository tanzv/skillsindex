# 2026-03-21 Skill 管理平台后端收口清单

## 1. 文档目的

本文件基于以下两份最新结论继续向前推进：

1. `docs/plans/2026-03-21-skill-management-backend-requirement-review.md`
2. `docs/design-requirements/overview/functional-module-completeness-summary.md`

目标不是重复判断“项目是什么”，而是明确：

1. 当前哪些差距已经被代码实现消化
2. 哪些差距仍然真实存在
3. 下一步应该按什么顺序收口
4. 每个模块收口到什么程度，才能从 `部分覆盖` 进入更稳定状态

## 2. 本轮收口原则

### 2.1 判定原则

一个模块不能仅凭“已有路由”认定为闭环。

至少需要同时满足：

1. 有明确后端接口
2. 有真实服务与模型承接
3. 成功路径与失败路径口径稳定
4. 审计、安全、权限边界可追溯
5. 验收矩阵可以直接落到自动化或可执行人工验证

### 2.2 与旧清单的差异

相较于 `docs/plans/2026-03-07-requirement-design-completeness-gap-checklist.md`，本轮有两个重要修正：

1. 若干被旧文档视为“未补齐”的能力，当前代码实际上已经落地
2. 当前更大的问题不是接口缺失，而是治理链路和发布闭环不完整

## 3. 已确认收口或明显前移的点

以下内容不应再按“完全缺失”处理：

### 3.1 账号安全基础能力

已经存在：

1. 登录失败限流
2. 忘记密码 request / confirm API
3. 服务端会话记录
4. 管理员状态变更、强制下线、密码重置

因此应修正的认知：

1. `身份与访问管理` 仍是 `部分覆盖`
2. 但原因不再是主链路缺失
3. 原因变为治理深度与最终交付模式未完全收口

### 3.2 审计字段基础模型

已经存在：

1. `request_id`
2. `result`
3. `reason`
4. `source_ip`

因此应修正的认知：

1. “AuditLog 字段不足”不能再按旧最小模型表述
2. 当前真正的问题应转向审计覆盖面、导出使用方式、运维证据链和自动化门禁

### 3.3 Skill 版本快照与回滚

已经存在：

1. `SkillVersion` 模型
2. create / visibility / delete / sync 自动生成版本
3. compare 能力
4. rollback / restore 生成新版本

因此应修正的认知：

1. 版本体系已经进入有效实现范围
2. 当前问题是它还没有与 sync run / job / audit 形成统一治理模型

### 3.4 组织治理与内容审核

已经存在：

1. 组织管理 API
2. 成员角色更新与移除
3. Skill 绑定 / 解绑组织
4. moderation case 模型
5. 举报与审核处理 API

因此应修正的认知：

1. 组织与审核不是纯目标态
2. 当前状态更准确地说是“后端能力已进入实现，但治理闭环未完成”

## 4. 当前真实缺口清单

### 4.1 P0: 同步治理统一链路

当前问题：

1. `sync policy` 仍偏 repository 默认策略别名
2. `sync-runs` 仍明显复用 sync jobs 视图
3. `job -> run -> version -> audit` 没有完全形成统一契约
4. 分页、筛选、重试、失败分布等治理接口不够强

收口目标：

1. 明确 job、run、version、audit 四层对象的边界
2. 让手动同步、定时同步、重试、回滚都进入同一条可追踪链路
3. 为管理端和验收矩阵输出稳定字段契约

完成定义：

1. `sync policy` 不再只是 alias 配置
2. `sync run` 有独立稳定的查询与详情模型
3. 每次成功同步都能追到版本与审计
4. 每次失败同步都能追到 run 与错误信息

### 4.2 P0: 异步任务治理收口

当前问题：

1. jobs 能力已存在，但与同步治理仍耦合松散
2. 状态机、幂等、重试策略、取消语义未在需求与实现侧完全统一
3. 发布验收里仍无法把任务治理视为真正闭环

收口目标：

1. 明确任务状态机
2. 明确 retry / cancel / duplicate submit 的行为
3. 将任务治理纳入 sync 链路，而不是平行存在

完成定义：

1. 任务状态流转固定
2. 幂等约束清晰
3. 同步任务与导入任务可以按统一观测维度筛查

### 4.3 P0: API Key scope 扩展与治理深化

当前问题：

1. 现有 scope 已安全收紧
2. 但 scope 覆盖范围仍然过窄
3. 缺少更强的运维治理动作，例如批量撤销、强制轮换、泄露处置

收口目标：

1. 扩展 scope 到更多公开只读能力
2. 为未来后台只读或集成型能力预留清晰命名空间
3. 补足运维处置动作与对应审计

完成定义：

1. scope 不再只覆盖搜索接口
2. 轮换与撤销具备更明确的治理规则
3. OpenAPI、测试和实现三者完全一致

### 4.4 P1: 账号治理最终闭环

当前问题：

1. 主链路已实现
2. 但用户自助停用仍未落地
3. 管理员密码重置仍是过渡式直接设新密码模型
4. 异常设备、地理位置、风险提醒等安全上下文未增强

收口目标：

1. 明确账号治理最终产品边界
2. 区分当前可上线基线与长期安全增强项
3. 让账号治理验收从“可用”进入“可发布”

完成定义：

1. 当前态与目标态口径清晰稳定
2. 密码重置交付模式明确
3. 关键失败路径和审计可稳定回归

### 4.5 P1: 组织治理规则收口

当前问题：

1. API 已有，但工作空间治理边界不够完整
2. 邀请、成员生命周期、平台救援等规则未完全细化
3. 需求和实现之间仍缺少最终验收口径

收口目标：

1. 把组织治理从“有能力”推进到“有稳定规则”
2. 明确平台角色与组织角色叠加边界
3. 明确成员变更、最后 owner、异常恢复的完整策略

完成定义：

1. 组织 owner / admin / member / viewer 行为矩阵稳定
2. 平台救援操作明确定义
3. 自动化与人工验收场景可执行

### 4.6 P1: 内容审核治理规则收口

当前问题：

1. 举报与审核动作已存在
2. 但申诉闭环未体现
3. case 状态、内容状态、审计结果未完全统一

收口目标：

1. 建立统一的 moderation case 生命周期
2. 区分“案件处理状态”和“内容当前状态”
3. 让审核与内容治理数据可以稳定追溯

完成定义：

1. resolve / reject / content action 的关系固定
2. 申诉是否在当前范围内得到正式定案
3. 审核动作对内容状态影响可验证

### 4.7 P1: 运维合规与发布证据

当前问题：

1. 相关 API 已存在
2. 但门禁自动化、监控真实性、恢复证据链仍不足
3. 当前很难按强发布口径证明系统已真正收口

收口目标：

1. 从“有运维接口”推进到“有可执行发布门禁”
2. 从“有审计导出”推进到“有可验证留存与恢复”
3. 从“有指标入口”推进到“有可信观测事实”

完成定义：

1. 发布前门禁可重复执行
2. 恢复演练与备份验证有可追溯记录
3. 审计导出与运维检查具备稳定证据格式

## 5. 推荐执行顺序

### 第一阶段：先打通主治理链路

1. 同步治理统一链路
2. 异步任务治理收口
3. Skill version 与 sync run 契约统一

原因：

1. 这是当前平台最核心的业务差距
2. 它直接决定“Skill 资产同步平台”是否真正成立

### 第二阶段：强化开放能力与安全治理

1. API Key scope 扩展与治理深化
2. 账号治理最终闭环

原因：

1. 这两块决定平台对外能力与账号安全基线
2. 它们已经有基础实现，继续推进成本可控

### 第三阶段：收口团队治理能力

1. 组织治理规则收口
2. 内容审核治理规则收口

原因：

1. 这两块更偏治理规则，不是基础能力空白
2. 应该在主业务链路稳定后再统一处理

### 第四阶段：补齐发布证据

1. 运维合规与发布门禁
2. 全量验收与回归矩阵固化

原因：

1. 这一步决定是否能够从“结构完整”进入“发布完整”

## 6. 建议拆解方式

建议后续任务拆解时，按以下格式执行：

1. 每个模块先补一页“当前实现与目标态契约差异”
2. 再补一页“接口与模型收口设计”
3. 最后补“测试与验收清单”

建议优先形成以下三组任务包：

1. `sync / job / version / audit` 统一治理任务包
2. `api key / public api / account security` 安全治理任务包
3. `organization / moderation / ops evidence` 治理闭环任务包

## 7. 一句话执行结论

下一步不应该再继续争论“后端有没有功能”，而应该直接进入“把现有能力收口成发布级治理闭环”的执行阶段。
