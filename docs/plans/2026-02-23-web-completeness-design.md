# SkillsIndex 最终 Web 完整性设计（头脑风暴记录）

日期：2026-02-23
主题：确保最终 Web 在用户端与管理后端的功能完整性

## 1. 目标

将“当前已实现能力”与“最终完整目标能力”统一到同一套需求框架，避免：

1. 仅覆盖当前代码、缺少未来完整性设计
2. 把目标态能力误判为已上线能力
3. 验收时缺少跨端（用户端+管理端）闭环标准

## 2. 备选方案

### 方案 A：仅维护当前实现覆盖

- 描述：只记录已上线功能，不引入目标态需求
- 优点：文档与代码一致性高，维护成本低
- 缺点：无法支持“最终完整 Web”目标，不利于版本规划

### 方案 B：双层覆盖模型（推荐）

- 描述：在同一文档体系中同时维护
  1) 已覆盖（当前实现）
  2) 目标态（待实现）
- 优点：
  1. 兼顾现状准确性与目标完整性
  2. 便于排期（需求 -> 实现 -> 验收）
  3. 便于对齐产品、研发、测试
- 缺点：需要多一个状态管理维度

### 方案 C：完全重写为目标态 PRD，再附实现映射

- 描述：先写完整目标 PRD，再反向映射当前实现
- 优点：战略视角更强
- 缺点：短期改造成本最高，且容易脱离现有代码细节

## 3. 选择与理由

选择方案 B（双层覆盖模型）。

理由：当前仓库已经具备较完整的“已实现功能需求”，最优策略是在此基础上补齐“目标态缺口”，通过状态标签完成一致性管理，而不是推翻重写。

## 5. 最终完整性定义

满足以下条件视为“最终完整 Web”达到目标：

1. 用户端：发现、认证、互动、账号中心（资料/密码/会话）
2. 管理端：导入、记录治理、用户治理、密钥治理、审计
3. 公共能力：OpenAPI、鉴权、钉钉接入、安全与可观测
4. 验收层：存在跨端 E2E 场景与专项验收（含 FR-ACC）
