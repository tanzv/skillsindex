# 2026-03-22 Skill 管理平台后端能力完整度清单

## 1. 文档目的

本文件用于在当前需求与代码基础上，给出一份更适合执行推进的后端能力完整度清单。

目标不是重复回答“项目是不是交易平台”，而是明确：

1. 当前平台后端的真实能力边界
2. 哪些模块已经达到较高完成度
3. 哪些模块已经有实现但仍未形成治理闭环
4. 哪些缺口仍然值得继续作为后续执行优先级

本文件结论基于：

1. `docs/plans/2026-03-21-skill-management-backend-requirement-review.md`
2. `docs/plans/2026-03-21-skill-management-backend-gap-closure-checklist.md`
3. `docs/plans/2026-03-22-sync-governance-unification-implementation-plan.md`
4. 当前 `backend/internal/models/**`
5. 当前 `backend/internal/services/**`
6. 当前 `backend/internal/web/**`

## 2. 平台定位结论

当前项目后端应明确定位为：

**企业内部 Skill 资产管理、同步、分发与治理平台后端**。

它不是：

1. 交易购买平台
2. 电商订单平台
3. 支付结算平台

它的核心后端职责是：

1. 统一接入 Skill 内容来源
2. 维护 Skill 数据、版本、可见性与组织归属
3. 提供同步调度、运行记录、重试取消、审计追踪
4. 提供账号、安全、组织、审核等治理能力
5. 对外提供公开查询 API 与 OpenAPI 契约

---

## 3. 完整度判定口径

一个模块不能仅因为“有页面”或“有路由”就认定为完整。

本清单采用以下判定口径：

### 3.1 已完成度较高

至少满足：

1. 有明确接口
2. 有真实后端服务与模型承接
3. 主成功路径可用
4. 主要权限边界已存在
5. 当前不属于平台主闭环风险点

### 3.2 已有能力但未闭环

表示：

1. 已有接口与后端能力
2. 主链路并非缺失
3. 但对象关系、治理规则、审计链路、失败路径或验收口径仍未完全统一

### 3.3 仍待完善

表示：

1. 能力存在明显规则空洞，或
2. 最终治理口径尚不稳定，或
3. 仍不足以作为强发布口径的完成态

## 5. 分模块结论

## 5.1 市场与发现

**状态：已完成度较高**

已有能力：

1. 公开搜索与检索链路
2. 分类、详情、时间线等发现能力
3. 对外查询定位清晰

判断：

1. 该模块服务于 Skill 资产发现
2. 产品方向稳定
3. 不属于本轮收口核心争议点

结论：

1. 不应再认定为“后端缺失”
2. 后续只需配合公开 API 契约持续稳定即可

## 5.2 互动系统

**状态：已完成度较高**

已有能力：

1. 收藏
2. 评分
3. 评论发布与删除
4. Skill 举报与评论举报

判断：

1. 主接口已存在
2. 互动链路具备基本闭环
3. 当前不是平台主治理缺口

结论：

1. 可以认定为后端主能力已落地

## 5.3 身份与访问管理

**状态：已有能力但未闭环**

已有能力：

1. 注册、登录、注销
2. 账号中心资料维护
3. 密码修改
4. 会话管理
5. 忘记密码 request / confirm
6. 管理员状态变更、强制下线、密码重置
7. 登录失败限流
8. 服务端会话记录
9. 基础审计字段

主要缺口：

1. 用户自助停用未形成稳定交付能力
2. 管理员密码重置模型仍偏过渡态
3. 设备、风险、异常会话上下文仍可增强

结论：

1. 不能说“未实现”
2. 更准确应定性为“主能力已实现，但最终安全治理闭环仍待补齐”

## 5.4 导入与记录治理

**状态：已完成度较高**

已有能力：

1. 手动导入
2. Zip 导入
3. 仓库导入/同步
4. SkillMP 导入/同步
5. visibility 切换
6. remote sync
7. delete

判断：

1. 入口与主服务链路已存在
2. records 治理不是空白
3. 当前短板不在“有没有导入能力”，而在后续统一治理链路

结论：

1. 可认定主链路完成度较高

## 5.5 同步治理

**状态：已有能力但未闭环**

已有能力：

1. `SyncPolicy` 一等模型已建立
2. `SyncJobRun` 已扩展为真实 run 合同
3. admin `sync-runs` 已有真实 handler
4. skill 维度 sync runs 能力已存在
5. repository batch 与 scheduler 已接入 governance
6. 单 skill remote sync 已接入 governance
7. `SyncGovernanceService` 已统一编排：
   - start
   - retry
   - complete
   - cancel

当前已明显前移的点：

1. `job -> run -> version -> audit` 已不再只是目标态描述
2. 关键主链路已经开始由统一 governance service 承接
3. sync-runs 不再只是简单 alias 语义

主要缺口：

1. 更强的分页、过滤、失败分布、治理报表仍可继续增强
2. policy 通用化与多来源策略管理仍可继续深化
3. 管理端与 OpenAPI 契约仍可继续强化稳定口径

结论：

1. 这是当前平台最核心但也最接近闭环的 P0 模块
2. 不能再说“同步治理没做”
3. 更准确是“核心收口已启动并取得实质进展，但尚未完全完成”

## 5.6 异步任务治理

**状态：已有能力但未闭环**

已有能力：

1. job 列表与详情
2. retry / cancel
3. 状态流转基础能力
4. 与同步链路的关联字段已存在
5. sync 类型 job 的 retry / cancel 已接入 governance

本轮新增确认：

1. `sync_repository` 与 `sync_skillmp` 类型 job
2. 当 job 已绑定 `sync_run_id` 时
3. retry / cancel 会优先进入 governance lifecycle
4. 旧非 sync 类型 job 保持兼容旧逻辑

主要缺口：

1. 全部异步任务类型的统一观测模型仍可继续收口
2. 幂等、重复提交、规则矩阵可继续外显化
3. 对外验收口径仍可更系统化

结论：

1. 异步任务治理不是缺失，而是正在从“有功能”向“有统一治理契约”演进

## 5.7 Skill 版本治理

**状态：已有能力但未闭环**

已有能力：

1. `SkillVersion` 模型
2. create / visibility / delete / sync 自动生成版本
3. compare
4. rollback / restore 生成新版本
5. `run_id` 关联字段已建立
6. remote sync 成功后的版本捕获已统一到 service 内部

主要缺口：

1. 版本治理与同步治理的查询口径还可继续增强
2. 面向管理端的版本审计视图仍可继续细化
3. 更稳定的链路报表与验收矩阵仍待完善

结论：

1. 版本体系已经进入有效实现范围
2. 当前问题不是“有没有版本”，而是“版本是否已完全纳入统一治理链路”

## 5.8 API Key 与公开 API

**状态：已有能力但未闭环**

已有能力：

1. OpenAPI 与 Swagger
2. API Key 鉴权
3. query / Bearer 双通道
4. 401 / 403 分流
5. deny-by-default scope 基线
6. 静态 key 对受保护接口已不再宽松放行

主要缺口：

1. scope 覆盖仍偏窄
2. 缺少更丰富的运维治理动作
3. 泄露处置、批量撤销、强制轮换等治理深度仍可继续增强

结论：

1. 安全边界较之前已明显收紧
2. 但离“完整治理型 API Key 系统”还有距离

## 5.9 组织治理

**状态：已有能力但未闭环**

已有能力：

1. 组织创建与基础管理
2. 成员角色变更与移除
3. Skill 绑定 / 解绑组织

主要缺口：

1. owner/admin/member/viewer 最终规则矩阵仍可更清晰
2. 邀请、成员生命周期、平台救援场景仍需系统化
3. 验收口径仍可继续固化

结论：

1. 组织治理是“已有能力，规则待收口”

## 5.10 内容审核治理

**状态：已有能力但未闭环**

已有能力：

1. moderation case 模型
2. 举报入口
3. 审核处理动作

主要缺口：

1. case 生命周期仍可进一步稳定
2. 内容状态与案件状态仍需更明确区分
3. 申诉闭环尚未体现为稳定范围结论

结论：

1. 审核治理已进入实现范围
2. 但还未达到强治理闭环标准

## 5.11 运维与发布证据

**状态：仍待完善**

已有能力：

1. 部分运维与审计接口
2. 一定程度的观测与导出入口

主要缺口：

1. 发布门禁自动化仍不够完整
2. 监控真实性与恢复证据链仍偏弱
3. 审计导出、恢复演练、备份验证等交付证据仍可增强

结论：

1. 这是当前“能运行”与“可强发布”之间的重要差距来源

---

## 6. 当前最高优先级排序

### P0

1. 同步治理统一链路继续收口
2. 异步任务治理继续与 sync chain 对齐
3. version / run / audit 查询与验收口径继续统一

### P1

1. API Key scope 深化
2. 账号治理最终闭环
3. 组织治理规则固化
4. moderation 生命周期规则固化
5. 运维发布证据增强

---

## 7. 本轮后的整体判断

当前平台后端不应再被表述为：

1. “没有后端能力”
2. “只有页面，没有接口”
3. “同步治理还没开始做”

更准确的表述应为：

1. **这是一个已经成型的企业内部 Skill 管理分发平台后端**
2. **大部分业务主能力已经实现**
3. **当前最重要的工作不是从零补接口，而是把关键治理链路彻底收口**
4. **其中同步治理与异步任务治理已经在本轮之前和本轮内取得实质性推进**

---

## 8. 建议的下一步执行方式

建议后续继续按以下顺序推进：

1. 继续细化 sync governance 查询、过滤、报表和契约输出
2. 把剩余与 sync chain 相关的边界动作全部统一到 governance service
3. 再进入 API Key、账号治理、组织治理和 moderation 的规则固化
4. 最后补发布门禁、审计导出、恢复证据等强交付能力
